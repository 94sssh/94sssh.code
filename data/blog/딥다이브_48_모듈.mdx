## 48.1. 모듈의 일반적 의미

> 애플리케이션을 구성하는 개별적 요소로서 재사용 가능한 코드 조각

모듈은 일반적으로 기능을 기준으로 파일 단위로 분리한다. 모듈이 성립하려면 자신만의 **파일 스코프(모듈 스코프)**를 가질 수 있어야 한다.

모듈은 캡슐화되어 다른 모듈에서 접근할 수 없고, 애플리케이션과 분리되어 존재한다. 그러나 완전히 분리시켜버리면 재사용이 불가능하므로 의미가 없다. 모듈은 재사용되어야 의미가 있으므로, `export`를 통해 필요한 자산에 한정해 선택적 공개가 가능하다.

모듈 사용자는 모듈이 공개한 자산 중 일부 또는 전체를 `import`를 통해 자신의 스코프로 불러들여 재사용할 수 있다.

모듈을 사용하면 코드의 단위를 명확히 분리해 애플리케이션을 구성할 수 있고, 재사용성이 좋아 개발 효율성과 유지보수성을 높일 수 있다.

## 48.2. 자바스크립트와 모듈

자바스크립트는 `import`, `export`를 지원하지 않았다. 기존에는 모든 자바스크립트 파일이 하나의 전역을 공유했었다. 이후,Node.js가 CommonJS를 채택하며 독립적인 파일 스코프를 가져 모듈을 지원하게 되었다.

## 48.3. ES6 모듈(ESM)

ES6에서 클라이언트 사이드 자바스크립트에서도 동작하는 모듈 기능을 추가했다. script 태그에 type="module" 어트리뷰트를 추가하면 로드된 자바스크립트 파일이 모듈로 동작한다. ESM임을 명확히 하기 위해 ESM 파일 확장자는 mjs 사용을 권장한다.

```html
<script type="module" src="app.mjs"></script>
```

### 48.3.1. 모듈 스코프

ESM은 독자적인 모듈 스코프를 갖는다. 모듈 내에서 선언한 식별자는 모듈 외부에서 참조할 수 없다.

### 48.3.2. export 키워드

모듈 내부에서 선언한 식별자를 다른 모듈이 재사용할 수 있게 하려면 `export`를 사용한다.

`export`는 선언문 앞에 사용한다. 대상을 하나의 객체로 구성해 한 번에 `export`할 수도 있다.

### 48.3.3. import 키워드

다른 모듈의 식별자를 모듈 스코프 내부로 로드하려면 `import`를 사용한다. `export`한 식별자 이름으로 `import`해야 하며 ESM이라면 파일 확장자를 생략할 수 없다.

모듈이 `export`한 식별자 이름을 일일이 지정하지 않고 하나의 이름으로 한 번에 `import`할 수도 있다. 이때 `import`되는 식별자는 as 뒤에 지정한 이름의 객체에 프로퍼티로 할당된다. 마찬가지로 식별자 이름을 변경하여 `import`할 수도 있다.

모듈이 하나의 값만 export한다면 default 키워드를 사용할 수 있고, 이 경우 이름 없이 하나의 값을 export한다. default를 사용하는 경우, var, let, const는 사용할 수 없다.
