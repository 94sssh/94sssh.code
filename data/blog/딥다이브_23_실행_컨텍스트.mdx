## 23.1. 소스코드의 타입

ECMAScript 사양의 소스코드 구분법

| 소스코드의 타입 | 설명                                                                                             |
| --------------- | ------------------------------------------------------------------------------------------------ |
| 전역 코드       | 전역에 존재하는 소스코드. 전역에 정의된 함수, 클래스 등의 내부 코드는 포함되지 않는다.           |
| 함수 코드       | 함수 내부에 존재하는 소스코드. 함수 내부에 중첩된 함수, 클래스 등의 내부 코드는 포함되지 않는다. |
| eval 코드       | 빌트인 전역 함수인 eval 함수에 인수로 전달되어 실행되는 소스코드                                 |
| 모듈 코드       | 모듈 내부에 존재하는 소스코드. 모듈 내부의 함수, 클래스 등의 내부 코드는 포함되지 않는다.        |

소스코드의 타입에 따라 실행 컨텍스트의 생성 과정과 관리 내용이 달라 4가지로 구분한다.

**1. 전역 코드**

전역 코드는 전역 스코프를 생성한다.  
전역 변수와 전역 함수를 바인딩하고 참조하기 위해 전역 객체와 연결되어야 하므로 전역 실행 컨텍스트가 생성된다.

**2. 함수 코드**

함수 코드는 지역 스코프를 생성한다.  
지역 변수, 매개변수 arguments 객체를 관리하며, 생성한 지역 스코프를 전역 스코프의 스코프 체인에 연결해야 하므로 함수 실행 컨텍스트가 생성된다.

**3. eval 코드**

eval 코드는 strict mode에서 독자적인 스코프를 생성한다.  
이를 위해 eval 실행 컨텍스트가 생성된다.

**4. 모듈 코드**

모듈 코드는 모듈별로 독립적인 모듈 스코프를 생성한다.  
모듈 코드가 평가되면 모듈 실행 컨텍스트가 생성된다.

## 23.2. 소스코드의 평가와 실행

1. 소스코드 평가 과정에서는 실행 컨텍스트를 생성하고 변수, 함수 등의 선언문을 먼저 실행한다. 생성된 변수나 함수 식별자는 실행 컨텍스트가 관리하는 스코프에 등록된다.

2. 소스코드 실행 과정(런타임)에서는 변수 할당 등 소스코드의 실행 결과를 실행 컨텍스트에 등록하고 관리한다.

## 23.3. 실행 컨텍스트의 역할

**1. 전역 코드 평가**

소스코드 평가 과정에서 변수 선언문, 함수 선언문 등 선언문만 먼저 실행하고, 생성된 전역 변수와 전역 함수는 전역 스코프에 등록되고, 전역 객체의 프로퍼티와 메서드가 된다.

**2. 전역 코드 실행**

런타임이 시작되어 전역 변수에 값이 할당되고 함수가 호출된다. 함수가 호출되면 전역 코드의 실행을 멈추고 함수 코드 실행으로 넘어간다.

**3. 함수 코드 평가**

함수가 호출되면 함수 코드 실행 준비를 한다. 매개변수와 지역 변수 선언문의 실행으로 생성된 매개변수와 지역 변수가 실행 컨텍스트가 관리하는 지역 스코프에 등록된다. 함수 내부에서 arguments 객체가 생성되고 this 바인딩이 결정된다.

**4. 함수 코드 실행**

함수 코드 평가 과정이 끝나면 함수 코드가 순차적으로 실행된다. 매개변수와 지역 변수에 값이 할당되고 메서드가 호출된다. 메서드 호출을 위해 스코프 체인을 검색하고, 메서드 실행이 종료되면 일시 중단되었던 전역 코드 실행으로 돌아가 실행을 계속한다.

위의 과정과 같이 **코드가 실행되려면 다음과 같이 스코프, 식별자, 코드 실행 순서 등의 관리가 필요한데, 이를 관리하는 것이 실행 컨텍스트다.** 실행 컨텍스트는 식별자를 등록하고 관리하는 스코프와 코드 실행 순서 관리를 구현한 내부 메커니즘으로, 모든 코드는 실행 컨텍스트를 통해 실행되고 관리된다.

1. 선언으로 생성된 모든 식별자(변수, 함수, 클래스 등)를 스코프를 구분하여 등록하고 상태 변화(식별자에 바인딩된 값의 변화)를 지속적으로 관리할 수 있어야 함.

2. 스코프는 중첩 관게에 의해 스코프 체인을 형성해 상위 스코프로 이동하며 식별자를 검색할 수 있어야 함.

3. 현재 실행 중인 코드의 실행 순서가 변경 가능하며 되돌아갈 수 있어야 함.

- 렉시컬 환경: 식별자와 스코프 관리
- 실행 컨텍스트 스택: 코드 실행 순서 관리

## 23.4. 실행 컨텍스트 스택(콜 스택)

전역 실행 컨텍스트, 함수 실행 컨텍스트 등 생성되는 실행 컨텍스트는 스택으로 관리되는데, 이를 **실행 컨텍스트 스택(콜 스택)** 이라고 부른다. 코드의 실행 흐름에 따라 실행 컨텍스트가 추가(`push`)되고 제거(`pop`)된다.

**1. 전역 코드의 평가와 실행**

전역 코드를 평가해 전역 실행 컨텍스트를 생성하고 실행 컨텍스트 스택에 푸시한다. 전역 변수와 전역 함수는 전역 실행 컨텍스트에 등록된다.

**2. 함수 코드의 평가와 실행**

함수의 호출로 함수 실행 컨텍스트를 생성하고 실행 컨텍스트 스택에 푸시한다. 이전 실행 컨텍스트의 실행이 일시 중단된다. 함수 코드가 실행되며 지역 변수에 값을 할당하거나 메서드를 호출한다.

**3. 코드의 복귀**

함수가 종료되면 함수 실행 컨텍스트를 실행 컨텍스트 스택에서 팝하여 제거하고 함수가 종료되며 코드의 실행 순서가 이전 코드로 돌아온다.

실행 컨텍스트 스택의 최상위 실행 컨텍스트는 항상 현재 실행 중인 코드의 실행 컨텍스트다.

## 23.5. 렉시컬 환경

렉시컬 환경은 스코프와 식별자를 관리한다. 렉시컬 환경은 두 개의 컴포넌트로 구성된다.

1. 환경 레코드  
   스코프에 포함된 식별자를 등록 및 바인딩된 값을 관리하는 저장소
2. 외부 렉시컬 환경에 대한 참조  
   외부 렉시컬 환경에 대한 참조는 상위 스코프를 가리키는데, 상위 스코프는 해당 실행 컨텍스트를 생성한 소스코드를 포함하는 상위 코드의 렉시컬 환경이다. 외부 렉시컬 환경에 대한 참조를 통해 스코프 체인을 구현한다.

## 23.6. 실행 컨텍스트의 생성과 식별자 검색 과정

```js
var x = 1
const y = 2

function foo(a) {
  var x = 3
  const y = 4

  function bar(b) {
    const z = 5
    console.log(a + b + x + y + z)
  }
  bar(10)
}

foo(20) // 42
```

### 23.6.1. 전역 객체 생성

전역 객체는 전역 코드가 평가되기 이전에 생성된다. 전역 객체에는 빌트인 전역 프로퍼티, 빌트인 전역 함수, 표준 빌트인 객체, 동작 환경에 따라 클라이언트 사이드 Web API 또는 특정 환경을 위한 호스트 객체를 포함한다.

### 23.6.2. 전역 코드 평가

전역 코드는 다음과 같은 순서로 평가된다.

1. 전역 실행 컨텍스트 생성
2. 전역 렉시컬 환경 생성  
    2.1. 전역 환경 레코드 생성

   - 2.1.1. 객체 환경 레코드 생성
   - 2.1.2. 선언적 환경 레코드 생성

   2.2. this 바인딩  
   2.3. 외부 렉시컬 환경에 대한 참조 결정

### 23.6.3. 전역 코드 실행

전역 코드가 실행되며 전역 변수에 x, y에 값이 할당되고 foo 함수가 호출된다. 식별자는 스코프가 다르면 같은 이름을 가질 수 있어 동일한 이름의 식별자가 다른 스코프에도 여럿 존재할 수 있다. 어느 스코프의 식별자를 참조할지 결정하는 것을 **식별자 결정**이라 한다. 식별자 결정을 위해 실행 중인 실행 컨텍스트에서 식별자 검색을 시작하는데, 해당 렉시컬 환경에서 식별자를 검색할 수 없으면 상위 스코프로 이동하여 검색한다.

전역 렉시컬 환경은 스코프 체인의 종점이므로 식별자가 검색되지 않으면 참조 에러(`ReferenceError`)를 발생시킨다.

### 23.6.4. foo 함수 코드 평가

foo 함수가 호출되면 함수 내부로 코드의 제어권이 이동되고 함수 코드 평가가 시작된다. 함수 코드는 다음과 같은 순서로 평가된다.

1. 함수 실행 컨텍스트 생성
2. 함수 렉시컬 환경 생성  
   2.1. 함수 환경 레코드 생성  
   2.2. this 바인딩  
   2.3. 외부 렉시컬 환경에 대한 참조 결정

### 23.6.5. foo 함수 코드 실행

런타임이 시작되어 foo 함수의 코드가 실행된다. 매개변수에 인수가 할당되며 지역 변수 x, y에 값이 할당되고 bar 함수가 호출된다. 이때 식별자 결정을 위해 실행 중인 실행 컨텍스트의 렉시컬 환경에서 식별자를 검색한다.

### 23.6.6. bar 함수 코드 평가

foo 함수 코드 평가와 동일하게 동작한다.

### 23.6.7. bar 함수 코드 실행

런타임이 시작되어 bar 함수의 코드가 실행된다. `console.log(a + b + x + y + z)`가 실행되는데, 순서는 다음과 같다.

1. console 식별자 검색  
   `console` 식별자를 찾아 스코프 체인을 검색한다.

2. log 메서드 검색  
   `console` 객체에서 프로토타입 체인을 통해 `log` 메서드를 검색한다.

3. 표현식 `a + b + x + y + z`의 평가  
   표현식을 평가하기 위해 각자의 함수 렉시컬 환경에서 식별자들을 검색한다.

4. console.log 메서드 호출

### 23.6.8. bar 함수 코드 실행 종료

더는 실행할 코드가 없어 함수 코드의 실행이 종료되며, 실행 컨텍스트 스택에서 bar 함수 실행 컨텍스트가 팝되고 foo 실행 컨텍스트가 실행 중인 실행 컨텍스트가 된다. 실행 컨텍스트가 제거되었다고 렉시컬 환경까지 소멸하는 것은 아니며, 누군가 참조하고 있다면 렉시컬 환경은 소멸하지 않는다.

### 23.6.9. foo 함수 코드 실행 종료

마찬가지로 함수 코드의 실행이 종료된다.

### 23.6.10. 전역 코드 실행 종료

남은 전역 코드가 없어 코드의 실행이 종료되며, 실행 컨텍스트 스택에 아무것도 남지 않게 된다.

## 23.7. 실행 컨텍스트와 블록 레벨 스코프

`let`, `const` 로 선언한 변수는 모든 코드 블록(함수, if 문, for 문, while 문, try/catch 문 등)을 지역 스코프로 인정하는 블록 레벨 스코프를 따른다.

if 문의 코드 블록 내에서 `let` 으로 변수가 선언되면, if 문의 코드 블록을 위한 블록 레벨 스코프를 생성해야 하므로 새롭게 렉시컬 환경을 생성해 기존 환경을 교체한다. for 문은 코드 블록이 반복 실행될 때마다 새로운 환경을 생성한다. 이때 새로운 환경의 외부 렉시컬 환경에 대한 참조는 문이 실행되기 전의 전역 환경을 가리킨다.
